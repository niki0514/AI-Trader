# AI Trader 产品规划

更新时间：`2026-03-10`  
适用范围：当前仓库中的策略流水线、API 服务与回测模块

## 1. 文档目标

这份文档用于回答三个问题：

- 这个项目现在已经做到了什么
- 下一阶段应该优先补什么
- 产品和工程边界分别在哪里

当前判断：项目已经从“单日脚本骨架”升级到“可运行的 MVP”，但仍处于面向研发和内部验证的阶段，不应误判为可直接生产上线的交易系统。

## 2. 当前产品定义

AI Trader 当前的核心定位是：

> 把一个交易日快照转成可解释、可回放、可查询的交易计划与组合结果，并在同一套契约上扩展回测与 API 服务能力。

当前产品包括三条主线：

1. 单日策略执行
2. 连续状态回测
3. 同步结果查询 API

## 3. 目标用户

### 3.1 策略研发 / PM

关注：

- 某日输入下系统会给出什么交易动作
- 风控为什么放行或拒绝
- 单日执行后组合会变成什么样

### 3.2 回测与数据工程

关注：

- 单日与多日是否共享同一套字段契约
- 回测是否能连续推进组合状态
- 输出是否足够支持 walk-forward 和参数敏感性分析

### 3.3 未来工作台或调度系统

关注：

- 是否能通过 API 获取持仓、计划、成交、日报
- 是否能稳定定位某个 `trade_date` 的最新结果
- 是否能按 run 追溯产物与阶段结果

## 4. 当前已实现能力

截至 `2026-03-10`，项目已经具备以下可用能力。

### 4.1 单日链路闭环

已实现从输入快照到完整产物输出的闭环，固定包含 7 个阶段：

1. `update_holding_actions`
2. `selector`
3. `analyst`
4. `decider`
5. `risk_guard`
6. `executor`
7. `reporter`

输出契约已稳定到以下主文件：

- `holding_actions_t.csv`
- `tech_candidates_t.csv`
- `ai_insights_t.csv`
- `orders_candidate_t.csv`
- `trade_plan_t.csv`
- `sim_fill_t.csv`
- `positions_t.csv`
- `nav_t.csv`
- `metrics_t.json`
- `risk_report_t.md`

同时保留阶段级 `json` 调试产物，方便排查。

### 4.2 有限降级

系统目前具备的是“有限降级”而不是“全链路降级”：

- `selector` 异常时可退化为空候选
- `risk_guard` 异常时可退回保守放行逻辑
- `executor` 异常时仍保底输出静态结果
- `update_holding_actions`、`analyst`、`decider`、`reporter` 采用 live LLM agent，未配置或调用失败时直接报错

产品意义是：规则层和执行层尽量保留可回放能力，但核心判断层不再伪造本地 fallback，避免产出“看似成功、实则失真”的结果。

### 4.3 状态化回测

回测已经不是简单循环调用单日脚本，而是支持：

- 按交易日生成子目录
- 汇总组合级 `NAV / fills / summary / report`
- 默认 `portfolio_state_carry=true`
- 连续传递 `cash / positions / initial_equity / peak_equity`

这意味着当前回测收益和回撤已经来源于连续组合状态，而不是“每日重置”的伪曲线。

### 4.4 同步查询 API

当前 API 已具备最小可接入能力：

- 可同步触发一次单日任务
- 可按交易日读取计划、成交、日报
- 可读取最新持仓和指定时间区间 NAV

这为后续接工作台、自动化脚本或调度服务提供了统一入口。

## 5. 当前产品边界

下面这些能力目前还不属于已完成范围：

- 没有工作台界面
- 没有用户体系与权限控制
- 没有任务队列与异步状态机
- 没有数据库持久化
- 没有真实券商下单
- 没有分钟级/盘口级成交回放
- 没有多策略、多账户、多租户隔离

因此当前版本应定义为：

> 面向研发验证和内部集成的本地化 MVP，而不是生产级交易平台。

## 6. 产品原则

### 6.1 文件优先

当前阶段继续坚持文件落盘，而不是提前引入数据库：

- 运行过程可审计
- 结果可直接人工核对
- 便于快速验证字段契约
- 便于回测日级回放

### 6.2 契约优先

短期内最重要的不是功能数量，而是口径稳定：

- 输入字段稳定
- CSV 列稳定
- 风控理由稳定
- API 返回结构稳定

### 6.3 明确失败优先于伪降级

当核心 agent 能力不可用时，优先保证：

- 不输出伪造的判断结果
- 失败原因清晰可定位
- 规则风控与执行结果仍可回放

### 6.4 同一流水线优先

单日、回测、API 都尽量复用一套流水线，而不是维护多份逻辑分支。  
这样有利于降低“线上结果”和“回测结果”之间的偏差。

## 7. 分阶段规划

### 7.1 P0：MVP 稳定化

状态：`已完成`

目标：

- 单日链路完整跑通
- 固化产物命名和字段
- 保留调试产物
- 支持规则层和执行层的有限降级

验收标准：

- 单日命令可直接运行
- 主产物缺失数为 `0`
- 风险报告和指标摘要可生成

### 7.2 P1：回测从“可跑”升级到“可信”

状态：`部分完成`

已完成：

- 回测入口可执行
- 支持连续状态推进
- 可生成组合级 summary 与 walk-forward report

下一步重点：

- 完善成本模型，细分佣金、税费、滑点
- 固化参数版本与样本区间记录
- 增加更可控的回测样本组织方式
- 为敏感性分析输出更标准的汇总结果

验收标准：

- 回测收益与持仓路径能复算
- 连续状态在跨日间不丢失
- 报告能解释主要收益、回撤和风控拦截

### 7.3 P2：API 从“能查”升级到“能接系统”

状态：`当前重点`

目标：

- 明确查询口径
- 保持返回结构稳定
- 为工作台与调度系统预留扩展空间

重点补强项：

- 增加接口文档和调用示例
- 明确 `single_day` 与 `backtest` 输出的检索优先级
- 统一错误码与错误消息
- 视接入需要增加任务查询接口

验收标准：

- 客户端不需要直读文件即可完成核心查询
- 相同日期多次运行时，“最新结果”口径可解释
- 文档足以支撑第三方接入

### 7.4 P3：服务化与生产化能力

状态：`未开始`

后续再考虑：

- 异步任务队列
- 任务状态持久化
- 鉴权与访问控制
- 数据库存储
- 监控与告警
- 多策略和多账户隔离

这一阶段只有在 P1、P2 足够稳定后才值得投入。

## 8. 建议优先级

基于当前代码现状，建议优先顺序如下：

1. 稳定 API 文档与查询口径
2. 提升回测真实度与成本模型
3. 增加参数版本、样本区间、实验记录
4. 为未来工作台补充更友好的聚合查询
5. 最后再引入数据库、鉴权、异步任务

## 9. 成功指标

### 9.1 系统侧

- 单日链路成功率 `>= 99%`
- 核心产物缺失数 `= 0`
- 阶段失败时仍能给出可解释输出
- 同一输入重复运行时结构稳定

### 9.2 风控侧

- 单票、行业、总仓位限制不穿透
- 每一笔风控拒绝或截断都可解释
- 回撤保护逻辑生效后结果符合预期

### 9.3 回测侧

- `cum_return / max_drawdown / sharpe_ratio / win_rate` 可稳定输出
- 连续状态回测结果可复算
- walk-forward 报告可直接用于研发复盘

### 9.4 集成侧

- 第三方仅通过 API 即可完成核心查询
- 文档能支持工作台或脚本快速接入

## 10. 主要风险

### 10.1 外部模型不稳定

live LLM 可能因为网络、密钥或速率限制失败，因此必须继续保留降级路径。

### 10.2 文件系统扩展性有限

随着运行次数增加，文件方式会面临：

- 检索效率下降
- 清理成本升高
- 并发写入协同难度上升

### 10.3 执行层真实度仍不够

当前是日频 paper execution，尚不能代表真实撮合结果。

### 10.4 缺少产品操作界面

当前结果更适合研发和脚本消费，不适合业务方直接使用。

## 11. 当前结论

截至 `2026-03-10`，AI Trader 已经完成了 MVP 阶段最关键的三件事：

- 单日链路闭环
- 连续状态回测
- 同步 API 查询

下一阶段不建议盲目扩展“大平台”，而应继续围绕三条主线推进：

- 让契约更稳定
- 让回测更可信
- 让 API 更易接入
